# toturial 编写规范（Shell 兼容层教程）

这份规范用于 `doc/toturials` 下所有后续文档，目标是让每篇
toturial 可快速上手、可追溯、可长期维护。

## 一、必须遵守的结构

每篇文档至少包含这三层结构：

1. **全局问题与边界**
   - 这篇文档解决哪类问题（用户问题 / 兼容缺口）
   - 运行时目标是什么（例如“让某套 workload 通过”）
   - 说明当前实现属于“兼容层”还是“完整实现”

2. **函数级说明**
   - 每个关键函数都要有三段：
     - 为什么加这个函数（动机）
     - 输入 / 处理流程（有序步骤）
     - 成功/失败行为（返回值与 errno）
   - 有边界条件时要单独列出来（例如 `EINVAL`、`EPIPE`、`ENOMEM` 等）

3. **兼容边界与删除条件**
   - 说明为何是 compat 方案
   - 写明什么条件下可删除/重构该方案
   - 留下与真实实现的差距说明

## 二、语言与风格

- 统一使用中文，尽量避免“只贴参数名/签名”的硬说明。
- 允许技术关键词保留英文（如 `syscall`, `compat`, `fd`），同时给中文解释。
- 要配**类比**：至少每章有 1 个“把复杂点讲清楚”的类比。
- 句子要短，避免连续的长段落堆砌。
- 不要大量使用“玄学术语”，优先“现象 -> 机制 -> 结果”叙述。
- 统一 markdown 层级，避免过深嵌套。

## 三、文件命名与顺序

- 章节文件使用 `NN-xxx.md`，前缀 `00`~`99` 保持可排序。
- 内容文件应只放一类主题，不要一份文件讲 3 个无关专题。
- 新文件必须在 `README.md` 中挂上章节序号和一句定位。
- 如涉及跨版本同步，文件名中仍然保留主标题含义，不加临时日期前缀。

## 四、函数条目模板

每个函数都按统一模板写（示例）：

### `sys_xxx(process, arg1, arg2...)`

#### 1) 场景
一句话说明这个系统调用在用户态什么场景被调用。

#### 2) 实现步骤
- 第 1 步：...  
- 第 2 步：...  
- ...

#### 3) 异常分支
- 输入无效：`EINVAL`  
- 地址非法：`EFAULT`  
- 不支持行为：`ENOSYS`/`EOPNOTSUPP`...

#### 4) Linux 差异说明
- 与 Linux 接近的地方  
- 与 Linux 不同且可接受的原因

#### 5) 类比（可选）
- 用现实场景快速讲清楚行为

## 五、更新动作清单（每次新增/大改文档都要执行）

- [ ] 在 `README.md` 更新章节列表或“阅读路径”说明（新增、移动、拆分时）
- [ ] 标注“主文件/变更范围”
- [ ] 写清 `起始提交 -> 目标提交`（至少标注主要提交点）
- [ ] 标注“删除条件”或“迁移条件”
- [ ] 列出“与其他章节的耦合点”（至少 2 条）

## 六、验收清单（写完后自检）

每篇文档完成后要检查：

- [ ] 关键函数是否都解释了返回错误码
- [ ] 是否说明了兼容边界
- [ ] 是否给了最少一个端到端链路（如 `sys_* -> UserProcess -> 用户态`）
- [ ] 是否能直接指导新同学“从哪里找问题”
- [ ] 在 `README.md` 是否可被检索到

## 七、版本与同步

- 当 `arceos` 代码同步到新 HEAD 时，先更新本规范再新增章节，避免后续风格漂移。
- 若新增实现与旧文档冲突，优先更新章节内容并在文中标注“变更点”。
- 所有 toturial 改动默认按“教学优先”组织，但不能掩盖实际行为差异。

## 八、引用约定

- 文档可引用代码路径时，优先使用相对路径。
- 文件内出现的关键路径建议采用短路径名（例如 `examples/shell/src/uspace.rs`）。
- 强制保持术语一致：`pipe` 不写成 `pipe() 管道口`、`shm` 不写成 `共享内存` 两套对照。
